# 跨平台项目设计规范参考

> 本规范从 the-flash 项目中提炼，可作为搭建其他跨平台项目的参考标准。

---

## 一、架构规范

### 1.1 前后端分离原则
- **客户端职责**：仅负责本地缓存和 UI 展示
- **后端职责**：统一管理业务逻辑和数据
- **通信方式**：RESTful API + JSON

### 1.2 数据同步策略
```
开始任务 → 下载模板到本地 → 本地执行 → 完成时一次性同步
```
- **离线优先**：网络中断时已执行的步骤不丢失
- **增量同步**：仅同步变更数据，减少带宽消耗
- **冲突处理**：以服务器数据为准，客户端保留本地副本供参考

### 1.3 API 设计统一规范
- **协议**：RESTful HTTP/HTTPS
- **数据格式**：JSON
- **响应结构**：
  ```json
  {
    "success": true,
    "message": "操作成功",
    "data": {}
  }
  ```
- **错误响应**：
  ```json
  {
    "success": false,
    "message": "错误信息",
    "code": "ERROR_CODE"
  }
  ```

## 二、代码规范

### 2.1 命名规范
| 类型 | 规范 | 示例 |
|------|------|------|
| API/DTO 字段 | snake_case | `user_id`, `created_at` |
| Kotlin/Java 变量 | camelCase | `userId`, `createdAt` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 类名 | PascalCase | `UserService`, `TaskRepository` |

### 2.2 时间格式
- **跨模块边界**：统一使用 ISO 8601 string
  ```
  2026-06-25T08:30:00Z
  2026-06-25T16:30:00+08:00
  ```
- **本地存储**：可使用时间戳（毫秒）
- **UI 展示**：根据用户时区格式化

### 2.3 文档语言
- **代码注释**：中文为主，技术术语保留英文
- **API 文档**：中英文双语
- **变量命名**：英文

### 2.4 验证规则
- 每次改动前必须做与风险匹配的验证
- 关键路径必须有单元测试覆盖
- UI 变更必须有截图对比

### 2.5 提交规范
- 只暂存当前任务明确涉及的文件
- 提交信息格式：`type(scope): description`
  - `feat`: 新功能
  - `fix`: 修复
  - `docs`: 文档
  - `style`: 格式
  - `refactor`: 重构
  - `test`: 测试
  - `chore`: 构建/工具

## 三、UI/UX 设计规范

### 3.1 设计风格（E-Ink 风格）
- **主色调**：黑白为主
- **对比度**：高对比度，确保可读性
- **阴影**：无阴影，扁平化设计
- **动效**：简洁，避免过度动画

### 3.2 颜色系统
```kotlin
// 主色调
Primary: #1B5E20 (深绿 - 安全/健康)
Secondary: #37474F (深灰蓝)
Background: #FAFAFA (浅灰白)
Surface: #FFFFFF (纯白)

// 状态色
Success: #2E7D32 (深绿)
Warning: #EF6C00 (橙色)
Error: #C62828 (深红)
Info: #1565C0 (蓝色)
```

### 3.3 字体规范
```kotlin
// 字体族
FontFamily: Outfit / Inter + 系统回退字体

// 字体大小（老年友好）
Display Large: 32.sp
Display Medium: 28.sp
Headline Large: 24.sp
Headline Medium: 20.sp
Title Large: 18.sp
Body Large: 16.sp (默认)
Body Medium: 14.sp
Label Large: 16.sp
```

### 3.4 间距与圆角
```kotlin
// 圆角
BorderRadius: 2.dp (极小圆角)

// 边框
Border: 2.dp 实线黑色，无阴影

// 间距
Padding: 8.dp, 16.dp, 24.dp, 32.dp
Margin: 8.dp, 16.dp, 24.dp, 32.dp
```

### 3.5 按钮规范
```kotlin
// 按钮样式
TextTransform: none (无大写转换)
FontWeight: 700 (粗体)
MinHeight: 48.dp (最小点击区域)
Padding: 16.dp 24.dp
```

### 3.6 卡片规范
```kotlin
// 卡片样式
Border: 2.dp 黑色边框
Shadow: 无
BorderRadius: 2.dp
Padding: 16.dp
```

## 四、老年友好设计规范

### 4.1 字体优先
- 默认字体：16sp
- 标题字体：24sp+
- 支持系统字体缩放（0.8x ~ 2.0x）

### 4.2 高对比度
- 文字与背景对比度 ≥ 4.5:1
- 关键信息使用高亮色
- 避免浅灰色文字

### 4.3 简洁布局
- 每个页面只展示核心信息
- 避免信息过载
- 使用卡片分组信息

### 4.4 大点击区域
- 按钮最小高度：48dp
- 间距充足，避免误触
- 关键操作按钮更大

### 4.5 明确反馈
- 操作后有明确的视觉反馈
- 加载状态清晰可见
- 错误信息易懂

### 4.6 无障碍支持
- 支持屏幕阅读器（TalkBack / VoiceOver）
- 语义化标签
- 键盘导航支持

## 五、安全规范

### 5.1 数据安全
- **凭证加密**：使用 AES-256 加密存储
- **传输加密**：所有 API 通信使用 HTTPS
- **最小权限**：只请求必要的设备权限
- **数据脱敏**：日志中不记录敏感信息

### 5.2 认证授权
- **JWT Token**：有效期 30 天，支持刷新
- **Token 存储**：安全存储（Keychain / Keystore）
- **自动刷新**：Token 过期前自动刷新
- **登出清理**：清除所有本地敏感数据

### 5.3 隐私保护
- **本地优先**：数据优先存储在本地
- **数据最小化**：只收集必要的用户数据
- **用户控制**：用户可以随时删除自己的数据
- **透明告知**：明确告知数据收集和使用方式

## 六、性能规范

### 6.1 网络优化
- **请求缓存**：合理设置缓存策略
- **请求合并**：批量请求减少网络往返
- **压缩传输**：启用 GZIP 压缩
- **超时设置**：合理设置超时时间

### 6.2 本地存储优化
- **索引优化**：为常用查询字段创建索引
- **数据清理**：定期清理过期数据
- **分页加载**：大数据集分页加载
- **懒加载**：按需加载数据

### 6.3 UI 渲染优化
- **列表懒加载**：使用 LazyColumn / LazyRow
- **图片缓存**：合理使用图片缓存
- **避免重组**：减少不必要的重组
- **动画优化**：使用硬件加速动画

---

## 七、测试规范

### 7.1 单元测试
- **覆盖率**：核心业务逻辑 ≥ 80%
- **命名规范**：`should_预期行为_when_条件`
- **测试隔离**：每个测试独立，不依赖外部状态
- **Mock 依赖**：使用 Mock/Stub 隔离外部依赖

### 7.2 UI 测试
- **关键路径**：覆盖主要用户流程
- **边界条件**：测试空数据、大数据、错误状态
- **无障碍测试**：验证屏幕阅读器支持
- **多设备测试**：不同屏幕尺寸适配

### 7.3 集成测试
- **API 测试**：验证接口请求和响应
- **数据库测试**：验证数据持久化
- **端到端测试**：验证完整用户流程

## 八、项目结构规范

### 8.1 KMP 项目结构
```
project/
├── shared/                    # 共享模块
│   └── src/
│       ├── commonMain/        # 共享代码
│       ├── androidMain/       # Android 特定代码
│       ├── iosMain/           # iOS 特定代码
│       └── jvmMain/           # JVM 特定代码
├── androidApp/                # Android 应用
├── iosApp/                    # iOS 应用
├── desktopApp/                # Desktop 应用
├── backend/                   # 后端服务
└── gradle/                    # Gradle 配置
```

### 8.2 包结构
```
com.terminator.shared/
├── model/                     # 数据模型
├── network/                   # 网络层
│   └── api/                   # API 接口
├── repository/                # 仓库层
├── storage/                   # 本地存储
├── di/                        # 依赖注入
└── util/                      # 工具类
```

### 8.3 命名约定
- **接口**：以 `I` 开头或使用 `-able` 后缀
- **抽象类**：以 `Abstract` 开头
- **实现类**：以 `Impl` 结尾
- **工厂类**：以 `Factory` 结尾
- **构建器**：以 `Builder` 结尾

## 九、依赖管理规范

### 9.1 版本目录
使用 `gradle/libs.versions.toml` 统一管理依赖版本：
```toml
[versions]
kotlin = "2.1.20"
ktor = "3.0.3"
compose = "1.7.3"

[libraries]
kotlin-test = { module = "org.jetbrains.kotlin:kotlin-test", version.ref = "kotlin" }

[plugins]
kotlin-multiplatform = { id = "org.jetbrains.kotlin.multiplatform", version.ref = "kotlin" }
```

### 9.2 依赖原则
- **最小依赖**：只引入必要的依赖
- **版本锁定**：使用精确版本，避免范围版本
- **定期更新**：定期更新依赖版本
- **安全审查**：检查依赖的安全漏洞

## 十、部署与发布规范

### 10.1 环境管理
- **开发环境**：本地开发，使用 Mock 数据
- **测试环境**：集成测试，使用测试数据
- **预生产环境**：预发布验证，使用生产数据副本
- **生产环境**：正式发布

### 10.2 发布流程
1. 代码审查通过
2. 所有测试通过
3. 版本号更新
4. 构建发布包
5. 部署到对应环境
6. 验证功能正常
7. 监控异常情况

### 10.3 版本号规范
使用语义化版本：`MAJOR.MINOR.PATCH`
- **MAJOR**：不兼容的 API 变更
- **MINOR**：向后兼容的功能新增
- **PATCH**：向后兼容的问题修复
